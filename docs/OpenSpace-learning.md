# HKUDS/OpenSpace 自动驾驶轨迹预测项目学习笔记

## 项目定位与核心思想

**OpenSpace** 是 HKUDS（香港大学数据科学实验室）开源的 **AI Agent 自我进化引擎**，核心目标是让 AI Agent 能够：

1. **从经验中持续学习** —— 每次任务执行后自动提取成功模式，修复失败案例
2. **集体智慧共享** —— Agent 之间可以共享进化后的技能，一人学会全体受益
3. **显著降低成本** —— 通过技能复用减少 46% Token 消耗，性能提升 4.2 倍

> "One Command to Evolve All Your AI Agents" —— 一条命令，进化你所有的 AI Agent

## 当前 AI Agent 的核心问题（OpenSpace 解决的痛点）

| 问题 | 描述 | OpenSpace 方案 |
|------|------|---------------|
| **大量 Token 浪费** | 每次都从零推理，重复消耗 Token | 复用成功模式，只更新增量部分 |
| **重复犯相同错误** | 失败模式没有记录，下次继续探索踩坑 | 自动记录错误，自动修复，一次解决终身受益 |
| **Skill 质量不可靠** | 工具 API 变化后 Skill  silently degrades | 多层质量监控，自动触发级联进化 |
| **知识孤岛** | 每个 Agent 独立，经验无法共享 | 云端技能社区，支持公开/私有/团队共享 |

## 核心架构设计

### 三大超能力

```
┌─────────────┐   ┌─────────────┐   ┌─────────────┐
│  🧬 自我进化 │   │  🌐 集体智慧 │   │  💰 Token效率 │
└─────────────┘   └─────────────┘   └─────────────┘
```

1. **🧬 自我进化** —— Skill 自动学习改进
   - AUTO-FIX：Skill 出错时即时就地修复
   - AUTO-IMPROVE：成功模式自动升级为更优版本
   - AUTO-LEARN：从实际使用中捕获高效工作流
   - 质量监控：全程跟踪 Skill 表现、错误率、成功率

2. **🌐 集体智慧** —— 个体进步转化为全体升级
   - 共享进化：一人改进 = 全体升级
   - 网络效应：更多 Agent → 更多数据 → 更快进化
   - 一键共享：上传下载只需要一条命令
   - 访问控制：支持公开/私有/团队三种模式

3. **💰 Token 效率** —— 越用越便宜
   - 不再重复劳动：复用成功方案而非从零开始
   - 任务越做越便宜：Skill 改进 → Token 消耗持续下降
   - 只做小幅更新：修复损坏部分，不需要全部重建
   - **实际数据：** 真实任务 4.2× 性能提升，Token 减少 46%

### 三种进化模式

| 模式 | 说明 | 使用场景 |
|------|------|----------|
| 🔧 **FIX** | 就地修复损坏/过时指令，同一 Skill 新版本 | 工具 API 变化，原有逻辑不工作 |
| 🚀 **DERIVED** | 从父 Skill 创建增强/专用版本，新目录共存 | 通用技能针对特定场景优化 |
| ✨ **CAPTURED** | 从成功执行中提取全新可复用模式，无父级 | 发现了从未有过的新模式 |

### 三个独立触发机制

| 触发器 | 触发时机 | 作用 |
|--------|----------|------|
| 📈 **执行后分析** | 每次任务完成后 | 分析完整执行记录，为相关 Skill 提出进化建议 |
| ⚠️ **工具退化检测** | 工具成功率下降时 | 质量监控找到所有依赖 Skill，批量触发进化 |
| 📊 **指标监控** | 定期扫描 | 扫描 Skill 健康指标（应用率/完成率/回退率），进化表现不佳者 |

### 级联进化（Cascade Evolution）

当任何组件退化（Skill 工作流 或 单个工具调用），**上游所有依赖的 Skill 自动触发进化**，维持系统级一致性。

## 基准测试结果（GDPVal）

在 220 个真实世界专业任务（GDPVal 数据集）上，使用相同骨干 LLM（Qwen 3.5-Plus）对比 ClawWork 基线：

| 指标 | OpenSpace | ClawWork 基线 | 提升 |
|------|-----------|---------------|------|
| **相对收入** | 4.2× | 1.0× | +320% |
| **价值捕获率** | 72.8% | ~17% | +55.8pp |
| **平均质量** | 70.8% | 40.8% | +30pp |
| **Phase 2 Token 用量** | 45.9% of Phase 1 | - | 减少 54.1% |

### 按类别分解的改进

| 类别 | 收入提升 | Token 减少 | 核心原因 |
|------|----------|------------|----------|
| 📝 文档与通信 | +3.3pp | -56% | document-gen-fallback 经历 13 个版本进化，结构和错误恢复接近全自动 |
| 📋 合规与表单 | +18.5pp | -51% | PDF 技能链（检查清单 → 排版 → 验证）进化一次，所有表单任务复用 |
| 🎬 媒体制作 | +5.8pp | -46% | 进化后的 Skill 编码了可用 ffmpeg 参数和回退策略，消除沙箱试错 |
| 🛠️ 工程 | +8.7pp | -43% | 协调类 Skill 在多样任务间通用迁移 |
| 📊 电子表格 | +7.3pp | -37% | 电子表格模式（公式/合并单元格/验证）跨领域通用 |
| 📈 战略与分析 | +1.0pp | -32% | 文档结构和多文件编排复用 |

### 进化产出统计（165 个 Skill）

**关键发现：** 大多数 Skill 聚焦于**工具可靠性和错误恢复**，而非特定任务知识。

| 用途 | 数量 | 教会 Agent |
|------|------|------------|
| 文件格式 I/O | 44 | PDF 回退/DOCX 解析/Excel 合并单元格，32/44 来自真实失败 |
| 执行恢复 | 29 | 分层回退：沙箱失败 → Shell → 写文件后运行 → heredoc，28/29 来自实际崩溃 |
| 文档生成 | 26 | 端到端文档流水线，`document-gen-fallback` 从 1 个进化出 13 个版本 |
| 质量保障 | 23 | 写后验证：检查行数/验证页数/校验公式，Phase 2 质量提升关键 |
| 任务编排 | 17 | 多文件追踪/ZIP 打包/零迭代失败检测，跨所有任务类型通用 |
| 领域工作流 | 13 | 特定领域流水线，如 SOAP 病历/音频制作/视频处理 |
| 网络与研究 | 11 | SSL/代理调试/搜索回退/JS 页面处理 |

## 与 guyong-juhuo 的对比与启发

### 架构理念对比

| 维度 | OpenSpace | guyong-juhuo |
|------|-----------|--------------|
| **核心目标** | 让任意 Agent 具备自我进化能力，共享技能 | 完整模拟特定个体，持续成长，最终思想超越人类 |
| **进化单元** | Skill（技能）| 判断 + 因果记忆 + 所有子系统 |
| **记忆模型** | SQLite 持久化版本 DAG（有向无环图）记录完整谱系 | 快慢双流 + causal_events.jsonl + causal_links.jsonl |
| **共享机制** | 云端社区，跨 Agent 共享 | 单个体成长，不强调跨个体共享 |
| **触发进化** | 执行后自动分析 + 工具退化 + 定期扫描 | 反馈系统闭环：人类反馈 → 更新因果记忆 → 更新自我模型 |

### OpenSpace 对 guyong-juhuo 的启发

1. **三级进化模式** —— FIX/DERIVED/CAPTURED 清晰分类，可以借鉴到因果记忆的更新策略：
   - FIX → 修正现有因果链接
   - DERIVED → 从现有因果衍生特定场景版本
   - CAPTURED → 捕获全新因果链接

2. **级联进化** —— 工具退化触发上游所有依赖 Skill 进化，这可以借鉴到**因果记忆的级联更新**：当某个基础因果链接被修正，所有依赖它的上层判断都应该被重新评估。

3. **质量监控** —— 全栈质量监控（Skill 层面/工具调用层面/代码执行层面），guyong-juhuo 可以在因果记忆层添加类似的质量追踪：每个因果链接记录应用次数/成功率/错误类型。

4. **Diff-Based 进化** —— 只生成最小化 diff 而非全量重写，节省 Token，这一点和我们当前做法一致，值得继续坚持。

5. **三种独立触发器** —— 除了反馈系统（类似执行后分析），可以添加：
   - **低置信度触发**：判断系统置信度低 → 标记为待探索 → 触发好奇心引擎
   - **周期性扫描**：定期扫描因果记忆中置信度低/成功率低的链接 → 触发重新学习

### 我们的设计优势

guyong-juhuo 在**个体模拟**方向比 OpenSpace 走得更深：

1. **完整十层架构** —— 从信息接收 → 判断 → 因果记忆 → 自我模型 → 好奇心 → 目标 → 情感 → 输出 → 行动 → 反馈，完整闭环
2. **因果记忆底座** —— OpenSpace 聚焦 Skill 层面，我们在判断和因果链路层面做了更细致的设计
3. **情感作为信号** —— 不模拟完整情感，只把情感作为决策信号，最早注入上下文，这是更务实的设计
4. **自我模型打通因果** —— 提醒不仅仅说"这里容易错"，还带上次错的前因后果，真正融入决策流程
5. **好奇心回流** —— 探索结果自动回流因果记忆，完整闭环

## 集成方式（给任何 Agent 使用 OpenSpace）

OpenSpace 设计为可插入任何支持 MCP（Model Context Protocol）的 Agent：

1. **添加 MCP 配置**：
```json
{
  "mcpServers": {
    "openspace": {
      "command": "openspace-mcp",
      "toolTimeout": 600,
      "env": {
        "OPENSPACE_HOST_SKILL_DIRS": "/path/to/your/agent/skills",
        "OPENSPACE_WORKSPACE": "/path/to/OpenSpace",
        "OPENSPACE_API_KEY": "sk-xxx (optional)"
      }
    }
  }
}
```

2. **复制两个宿主 Skill**：
   - `delegate-task` —— 教会 Agent 何时委托任务给 OpenSpace
   - `skill-discovery` —— 教会 Agent 如何搜索发现 Skill

3. **完成** —— Agent 现在可以自动使用 OpenSpace 的自我进化能力

支持三种传输模式：
- **stdio**：最简单，嵌入 Agent 进程
- **SSE**：独立服务，远程接入
- **streamable HTTP**：独立服务，支持流式响应

## 关键设计决策点评

| 决策 | 点评 |
|------|------|
| **Skill 不是静态文件，是活的实体** | 非常赞同，这是对当前 Skill 系统的重大改进。我们现在也是这么认为：guyong-juhuo 的每个子系统都是可成长的。 |
| **三种进化模式清晰分离** | 分类清晰，容易落地，值得借鉴。FIX/DERIVED/CAPTURED 覆盖了所有进化场景。 |
| **级联进化维护一致性** | 这个洞见很重要，在复杂依赖系统中，如果只修复叶子节点不修复上游，一致性会慢慢丧失。 |
| **Diff-Based 节省 Token** | 正确，全量重写浪费 Token 还容易引入新错误。 |
| **三种独立触发器** | 多层防御，确保问题不会被漏掉。比只有"执行后分析"更健壮。 |
| **165 个进化 Skill 中大多数是错误恢复** | 这个发现很有意思，真实世界中大部分进化来自解决失败，而非发现新知识。验证了"失败比成功更有学习价值"。 |
| **支持三种传输模式适配不同场景** | 工程上考虑周到，从嵌入式到独立服务都覆盖。 |

## 路线图

OpenSpace 当前已经解决了**"how to do X"**的进化，下一步方向：

- [ ] **看板式编排** —— 具备 Skill 感知调度的共享任务板，调度策略本身也能进化
- [ ] **协作模式进化** —— 从已完成任务中捕获并改进分解、交接、优先级策略
- [ ] **角色涌现** —— Agent 通过实践发展角色画像，而非依赖配置
- [ ] **跨团队模式迁移** —— 一个团队发现的协调模式可通过云端供其他团队使用

## 总结

OpenSpace 是一个非常成熟的 **AI Agent 自我进化引擎**，核心贡献在于：

1. **清晰的进化模式分类**（FIX/DERIVED/CAPTURED）
2. **多层触发机制**确保问题不被遗漏
3. **级联进化**维持系统级一致性
4. **实测数据验证** —— 在真实任务上确实带来 4.2× 性能提升和 46% Token 减少
5. **工程落地完整** —— 支持 MCP 插入任意 Agent，三种传输模式，本地仪表盘

对我们 guyong-juhuo 项目最有价值的启发是**级联更新**和**质量监控**的思路，可以整合到因果记忆系统中，让整体更健壮。
